# What LedgerProof EU does — and how to use it

**Audience:** Any institution, regulator, investor, or partner asking "what exactly does this do, and how would I use it?"
**Tone:** Sober institutional — no marketing voice. Ready to share verbatim.

---

## The problem, in one paragraph

The EU AI Act Article 50 — enforcing **August 2, 2026** — requires that
deployers of AI systems disclose when content is AI-generated, in a
**machine-readable** form. Today there is no working answer: C2PA
metadata gets stripped by social platforms ~95% of the time, watermarks
are removable, file metadata is editable, vendor-hosted attestation logs
can be re-written. A German insurance company that uses GPT-4o to draft
policy summaries and then has to defend that work in a BaFin audit 18
months later has no cryptographic way to prove what was generated by AI,
by whom, on what date, without a single point of failure. That's the gap.

## What LedgerProof EU is

A **calendar operator** — an HTTP API hosted in Frankfurt that accepts
Ed25519-signed receipts referencing AI-generated content, batches them
daily, and commits the Merkle root of each batch into a Bitcoin mainnet
OP_RETURN transaction. The receipts themselves live in an EU-resident
PostgreSQL; only the hash root goes on-chain. Any party — regulator,
court, counterparty — can independently verify a receipt against
Bitcoin, the deployer's public key, and the original document, with
zero trust in LedgerProof required beyond the moment of receipt issuance.

## How a deployer (institution) uses it

### One-time setup
1. Sign up — get an API key + register an Ed25519 public key
2. Decide deployer identity: their LEI (Legal Entity Identifier), or
   EUID, or VAT number

### Every time the institution uses AI to generate regulated content

1. The deployer's AI system generates a document.
2. The deployer's backend computes SHA-256 of the document.
3. Builds a canonical entry referencing the hash:

   ```
   content_type:  "ai/article-50/v1"
   content: {
     ai_system_id:    "openai/gpt-4o/2024-11-20",
     deployer_id:     "LEI:5493001KJTIIGC8Y1R12",
     deployer_name:   "Acme Insurance GmbH",
     deployer_country: "DE",
     content_category: "AI_ASSISTED_DOCUMENT",
     artifact_hash:   "a1b2c3...",
     artifact_bytes:  48291
   }
   ```

4. Signs the entry with their Ed25519 private key.
5. `POST https://api-eu.ledgerproofhq.io/v1/publish`
6. Receives back: sequence number + entry_hash.
7. Stores the receipt alongside the document.

Total time: **less than 1 second**.

24 hours later, the Foundation anchors the day's batch on Bitcoin. The
deployer's receipt is now retroactively timestamped against the
confirming block — they can fetch the Merkle inclusion proof at
`GET /v1/receipts/{seq}` whenever they want.

**The deployer's document never touches LedgerProof infrastructure.**
Only the hash.

## How a regulator (EU AI Act enforcement) verifies a receipt

Scenario: BaFin is auditing Acme Insurance's Article 50 compliance.

1. BaFin asks Acme: "show me the Article 50 receipts for every
   AI-generated policy summary you issued in Q3 2026."
2. Acme provides:
   - The policy summary PDF
   - The LPR receipt (JSON) referencing its hash
3. BaFin's auditor performs **four independent checks**:

   **a. Hash the PDF → SHA-256 → matches receipt.artifact_hash.**
   *"The document is what the receipt says it is."*

   **b. Verify the Ed25519 signature against Acme's registered public key.**
   *"Acme really attested to this, not a forger."*

   **c. Read the Bitcoin transaction at receipt.anchor.txid from any
   Bitcoin node — extract the Merkle root from its OP_RETURN.**
   *"Bitcoin proves this root existed at block height N mined on date D."*

   **d. Verify the Merkle inclusion proof in the receipt against that root.**
   *"This specific receipt was in that anchored batch."*

4. All four checks pass → BaFin has cryptographic proof that Acme
   committed to "this document, by this AI system, on this date" at a
   time no later than block N's timestamp.

   Acme cannot have backdated it. Acme cannot deny it. LedgerProof
   cannot have forged it. Bitcoin cannot lie about block N's existence.

The regulator **never needs to trust LedgerProof** for the proof. They
need to trust Bitcoin, SHA-256, and Ed25519. All three are public,
peer-reviewed, decades-old, and standardized at NIST / IETF level.

## What exactly gets cryptographically proven

| Property | How it's proven |
|---|---|
| **Authenticity** — the deployer made this attestation | Ed25519 signature against deployer's published public key |
| **Integrity** — the document hasn't been altered since attestation | SHA-256 hash comparison |
| **Timestamp** — the attestation existed no later than block N | Bitcoin proof-of-work + Merkle inclusion proof |
| **Identity** — which AI system, which deployer, which jurisdiction | Validated server-side at issuance, signed by deployer, anchored to Bitcoin |
| **Non-repudiation** — the deployer cannot later deny attesting | Signature + on-chain anchor combination is mathematically impossible to forge |
| **Verifier independence** — no trust in LedgerProof required after issuance | All proofs are checkable with public Bitcoin data + standard cryptographic libraries |

## What's stored where (data residency / GDPR clarity)

| Data | Location | Personal data? | Retention |
|---|---|---|---|
| AI-generated document itself | **Deployer's own systems only** | Maybe — deployer's responsibility | Deployer's policy |
| Document SHA-256 hash | EU Postgres (Frankfurt) | No — it's a one-way hash | Until GDPR erasure request |
| Deployer LEI, AI system ID, signed entry | EU Postgres (Frankfurt) | No — LEI is legal-entity | Same |
| Merkle root of daily batch | Bitcoin mainnet OP_RETURN | No — it's a 32-byte hash of hashes | Permanent (by design) |

When an EU data subject demands GDPR Article 17 erasure:
`DELETE /v1/entries/:seq` nulls the personally-identifiable content
fields in the EU database but keeps the cryptographic chain intact. The
Bitcoin anchor doesn't change (it's a hash of hashes — contains no PII)
but the receipt's content is no longer recoverable through LedgerProof.

## Where this matters commercially

Every regulated EU enterprise that deploys an AI system that generates
document-like content — insurance summaries, legal drafts, customer
communications, medical records, KYC packets, ESG reports — has an
August 2 problem with no other working answer at protocol level.

- C2PA (Content Credentials) is **necessary** (rich metadata) but
  **insufficient** (gets stripped by platforms 95% of the time).
- Microsoft Agent Governance Toolkit is **necessary** (governance
  posture) but **insufficient** (no immutable anchor).
- LedgerProof is the **immutable anchor layer** that makes the other
  two stacks regulatorily defensible.

That is the wedge for institutional adoption, the wedge for Frank's
payment-systems network, the wedge for the TVP investment thesis, and
the wedge for the first 10 EU pilot customers.
