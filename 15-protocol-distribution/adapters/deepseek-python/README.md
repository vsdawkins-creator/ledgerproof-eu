# ledgerproof-deepseek

LedgerProof adapter for **DeepSeek**, which exposes an OpenAI-compatible REST API
at `https://api.deepseek.com`. This package wraps the official
[`openai` Python SDK](https://github.com/openai/openai-python) configured for that
endpoint and emits signed EU AI Act **Article 50** transparency receipts as a
side channel — your DeepSeek response is returned unmodified.

```
pip install ledgerproof-deepseek
```

## Why DeepSeek matters for Article 50

DeepSeek ships three distinct surfaces that map to different Article 50
obligations:

- `deepseek-chat` — general conversational inference (Art. 50(1) chatbot).
- `deepseek-reasoner` (a.k.a. R1) — chain-of-thought reasoning model whose
  intermediate "thinking" trace is exposed alongside the final answer. The
  reasoning trace itself is content generated by an AI system; deployers who
  surface or rely on the trace need an Article 50(1)/(2) binding that covers
  both the trace and the answer.
- `deepseek-coder` — code generation. Code emitted by an AI system that is
  shipped into production, into a customer-visible artifact, or into a public
  repository is Article 50(2) "generated content" and benefits from a
  verifiable provenance receipt.

This adapter ships four schemas tuned to those DeepSeek-specific scenarios:

| Schema                       | Article basis | Use when                                          |
|------------------------------|---------------|---------------------------------------------------|
| `chatbot_session/v1`         | Art. 50(1)    | Standard conversational `deepseek-chat` inference |
| `generated_content/v1`       | Art. 50(2)    | Synthetic content output (text/image/audio/video) |
| `reasoning_trace/v1`         | Art. 50(1)    | `deepseek-reasoner` with chain-of-thought trace   |
| `code_generation/v1`         | Art. 50(2)    | `deepseek-coder` code-generation inference        |

## 5-minute quickstart

```python
from deepseek_ledgerproof import LedgerProofDeepSeek

# Defaults: base_url=https://api.deepseek.com, api_key=$DEEPSEEK_API_KEY
client = LedgerProofDeepSeek(
    deployer_id="urn:eu:deployer:acme-bank-de",
    regulatory_context={"schema": "chatbot_session/v1", "jurisdiction": "EU"},
)

resp = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "Hello"}],
)

print(resp.choices[0].message.content)
# A signed receipt has been emitted to the configured emitter
# (stdout LogEmitter by default).
```

## Three integration patterns

### 1. Client wrapper (recommended)

```python
from deepseek_ledgerproof import LedgerProofDeepSeek, LedgerProofAsyncDeepSeek

client = LedgerProofDeepSeek(deployer_id="urn:eu:deployer:acme")
async_client = LedgerProofAsyncDeepSeek(deployer_id="urn:eu:deployer:acme")
```

Works with both **non-streaming** (`stream=False`) and **streaming**
(`stream=True`) chat completions. Streaming uses incremental SHA-256
over each chunk (constraint **C6**).

### 2. Decorator

```python
import os
from openai import OpenAI
from deepseek_ledgerproof import lpr_track

client = OpenAI(
    api_key=os.environ["DEEPSEEK_API_KEY"],
    base_url="https://api.deepseek.com",
)

@lpr_track(deployer_id="urn:eu:deployer:acme")
def ask(question: str, *, messages):
    return client.chat.completions.create(model="deepseek-chat", messages=messages)
```

### 3. Manual emission

```python
import os
from openai import OpenAI
from deepseek_ledgerproof import emit_receipt

client = OpenAI(api_key=os.environ["DEEPSEEK_API_KEY"], base_url="https://api.deepseek.com")
resp = client.chat.completions.create(
    model="deepseek-chat",
    messages=[{"role": "user", "content": "hi"}],
)
emit_receipt(resp, deployer_id="urn:eu:deployer:acme")
```

## Reasoning trace (`deepseek-reasoner`)

DeepSeek-R1 returns a `reasoning_content` field on the assistant message in
addition to the standard `content`. The reasoning trace is itself AI-generated
text and falls inside the Article 50 transparency surface for any deployer who
surfaces it to users (or makes downstream decisions based on it).

```python
from deepseek_ledgerproof import LedgerProofDeepSeek

client = LedgerProofDeepSeek(
    deployer_id="urn:eu:deployer:acme",
    regulatory_context={"schema": "reasoning_trace/v1"},
)
resp = client.chat.completions.create(
    model="deepseek-reasoner",
    messages=[{"role": "user", "content": "Prove sqrt(2) is irrational."}],
)
# The receipt records both:
#   - response_sha256       — hash over the final answer text
#   - reasoning_sha256      — hash over the chain-of-thought trace
# so an auditor can verify the binding between the trace surfaced to the user
# and the final answer, without the deployer disclosing the raw trace text.
```

The adapter automatically extracts `reasoning_content` from
`response.choices[].message.reasoning_content` (the field DeepSeek's
OpenAI-compatible API uses for the R1 trace). If your downstream pipeline
discards the trace, it is still recorded by hash in the receipt.

## Code generation (`deepseek-coder`)

```python
from deepseek_ledgerproof import LedgerProofDeepSeek

client = LedgerProofDeepSeek(
    deployer_id="urn:eu:deployer:acme",
    regulatory_context={
        "schema": "code_generation/v1",
        "programming_language": "python",
    },
)
resp = client.chat.completions.create(
    model="deepseek-coder",
    messages=[{"role": "user", "content": "Write a function to reverse a string."}],
)
```

The `code_generation/v1` schema records `programming_language` and
`content_modality="code"`, suitable for use as Article 50(2) provenance
metadata when generated code is committed to a repository or shipped to an end
user.

## Custom emitters

```python
from deepseek_ledgerproof import LedgerProofDeepSeek, WebhookEmitter, MultiEmitter, LogEmitter

client = LedgerProofDeepSeek(
    deployer_id="urn:eu:deployer:acme",
    emitter=MultiEmitter([
        LogEmitter(),
        WebhookEmitter(url="https://receipts.acme.example/ingest"),
    ]),
)
```

## Scope and discipline (C1-C8)

This adapter implements the LedgerProof protocol with the following discipline:

- **C1 - No regulator endorsement.** This package is not endorsed by the European
  Commission, EU AI Office, or any national supervisory authority. It does **not**
  provide an Article 40 presumption of conformity.
- **Not endorsed by DeepSeek.** This is an independent adapter built on top of
  DeepSeek's publicly documented OpenAI-compatible API. It is not affiliated
  with, sponsored by, or endorsed by DeepSeek (Hangzhou DeepSeek Artificial
  Intelligence Basic Technology Research Co., Ltd. / DeepSeek AI). "DeepSeek",
  "DeepSeek-R1", and "DeepSeek-Coder" are trademarks of their respective
  owner; used here for descriptive interoperability purposes only.
- **C4 - Offline verification.** Receipts are verifiable offline against the
  published protocol public key and Bitcoin OP_RETURN anchor. The adapter does
  not phone home.
- **C6 - Stream-aware signing.** Streaming completions are hashed incrementally
  so the receipt covers the full reconstructed text. The hasher never buffers
  the entire response body in memory.
- **C7 - Side-channel only.** This adapter never modifies the DeepSeek response.
  Receipts emit through a separate emitter (log/stderr/webhook/queue/multi), and
  emitter failures never propagate into the caller path.

### What this adapter does NOT do

- Does **not** address Article 9 (risk management), Article 10 (data governance),
  Article 13 (transparency to deployers), Article 15 (accuracy/robustness/
  cybersecurity), or Article 72 (post-market monitoring).
- Does **not** claim Article 40 presumption of conformity.
- Does **not** claim DeepSeek endorsement or affiliation.
- Does **not** evaluate the quality, accuracy, or jurisdiction-of-training of
  the underlying DeepSeek models. The deployer is responsible for the
  contractual and regulatory posture of using a non-EU model provider in their
  EU deployment.
- Provides the **Article 50 evidence layer only** for users of DeepSeek via the
  OpenAI-compatible API.

## Known limitations (v0.1)

- Ephemeral in-process Ed25519 keys only. HSM-backed signing (AWS KMS, GCP KMS,
  Azure Key Vault, YubiHSM2) lands in v0.2.
- DeepSeek-R1 reasoning trace extraction depends on the
  `message.reasoning_content` field name as exposed by DeepSeek's
  OpenAI-compatible API. If DeepSeek changes the field, the trace will be
  recorded as an empty hash until the adapter is updated; the rest of the
  receipt remains valid.
- The adapter does not inspect generated code for licensing or vulnerability
  issues — `code_generation/v1` records the provenance binding only; deployers
  are responsible for downstream SCA / SAST.
