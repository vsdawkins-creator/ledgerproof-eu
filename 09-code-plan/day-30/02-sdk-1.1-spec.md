# SDK 1.1 — Specification

**Scope:** Python (`ledgerproof`) and TypeScript (`@ledgerproof/sdk`) SDKs at the 1.1 release. The 1.1 release is **the production-hardening release** — the one that runs under real customer load through Article 50 enforcement. No new protocol features; all new features are operational.

**Owner:** Founder → Senior Product Engineer (hire by Day 30)
**Target release date:** July 18, 2026 (T-15 from enforcement)
**Versioning:** Strict SemVer. 1.x must remain backwards-compatible at the API surface; behavior changes are documented in CHANGELOG.

---

## What's in 1.1

### Local queue fallback (P0)

If the operator is unreachable, the SDK must not block the inference path. The SDK enqueues receipts locally (SQLite-backed by default; pluggable backend) and retries with exponential backoff + jitter. The SDK exposes a `queue_status()` method that returns depth, oldest entry age, and last-success timestamp.

**API:**
```python
client = ledgerproof.Client(
    api_key=...,
    fallback_queue=ledgerproof.SqliteQueue(path="/var/lib/lpr/queue.db"),
    fallback_max_age_seconds=86400,  # alert after queue depth > 24h
)
status = client.queue_status()
# { depth: 42, oldest_age_seconds: 312, last_success_at: ... }
```

**Behavior:** If `fallback_queue` is None (default for v1.0 compatibility), the SDK raises on operator unavailability — preserves v1.0 behavior. Customers opt in to fallback explicitly.

**Tests:** Operator-unreachable simulation; queue-recovery on restart; queue-drain rate matches operator throughput.

### Retry policy with jitter (P0)

Idempotency keys generated client-side per receipt. Operator deduplicates by key. SDK retries on 5xx and on connection errors. Exponential backoff capped at 60 seconds; max retries configurable (default 5). Full jitter strategy.

**Default backoff:** `min(60, 0.5 * 2^attempt) + random.uniform(0, 0.5)`

### Streaming receipt issuance (P1)

For high-throughput pipelines, batch receipt issuance API. SDK collects N receipts or T milliseconds (whichever first), submits as a batch to the operator's batch endpoint.

```python
async with client.batch(max_size=100, max_wait_ms=200) as batch:
    for output in stream:
        await batch.issue(...)
```

**Operator-side:** Batch endpoint accepts up to 1000 receipts per call; per-receipt validation; partial-success response.

### LangChain integration smoothed (P1)

`langchain-ledgerproof` 1.1 must work with LangChain 0.3.x and any 0.4.x preview. Specifically:

- `LedgerProofCallbackHandler` works with both legacy and LCEL chains
- Receipt is attached to LangSmith trace when LangSmith is in use
- `with_ledgerproof()` chain builder method
- Async + sync paths tested

### Telemetry & opt-out (P0)

SDK reports anonymous usage telemetry (Python version, OS, SDK version, anonymized customer ID) unless `telemetry=False` is passed to the client constructor. Telemetry payload is documented at `docs.ledgerproofhq.io/telemetry`. Default is opt-in for first 30 days post-1.1 release to gather hardening signal; switches to opt-out by 1.2.

**Privacy review required before release.** Pentest scope explicitly tests telemetry payload for accidental PII.

### Strict schema validation at parse time (P0)

The SDK enforces GDPR safety net at the SDK boundary, before any value reaches the operator:

- `deployer_id`: rejects strings matching email regex; rejects strings > 256 chars
- `reviewer_role`: rejects email regex; enum-validated against published role list or "custom-<sluglike>"
- `review_rationale`: rejects email regex; rejects strings > 8KB
- `data_lineage`: rejects email regex in any string-typed field

**Tests:** Each rejection path has a unit test; reject-list maintained in `ledgerproof._validators` and shared between Python + TS via codegen.

### Configurable Bitcoin anchor SLO advisory (P2)

Customer can configure expected anchor latency; SDK emits a callback when measured latency exceeds threshold. Used by customers integrating LedgerProof into their own observability stack.

### Multi-region operator selection (P2)

Once US operator is up (Day 60), SDK supports region preference: `region="eu"` (default), `region="us"`, or `region="auto"` (DNS-based selection). For 1.1, only `region="eu"` is functional; the API surface is shipped now so 1.2 can light up multi-region without API change.

### TypeScript SDK feature parity (P0)

Every Python feature above must ship in TS at the same release. Codegen for schema validators is the mechanism — schemas defined in a single `schemas/` repo, Python and TS generated from it.

### Documentation updates (P0)

- Migration guide 1.0 → 1.1
- "Production checklist" page (queue, retry, telemetry, SLOs)
- Code examples for OpenAI, Anthropic, Azure OpenAI, Bedrock, Vertex AI

## Release process

1. Tag `v1.1.0-rc.1` on Day 30 - 7 (July 18 if Day 30 = July 25)
2. Beta cohort of 5 customers (existing pilots) installs RC for 5 business days
3. Tag `v1.1.0` if no Critical/High issues
4. PyPI + npm publish via OIDC trusted publishers (no long-lived tokens)
5. GitHub release with SBOM artifact, checksums, signed tag
6. Slack + email notification to customers

**Rollback:** Customers who hit a critical regression are instructed to `pip install ledgerproof==1.0.X` or `npm install @ledgerproof/sdk@1.0.X`. The 1.0.x line remains supported through Day 90.

## Open questions (must close before release)

1. **Fallback queue default backend.** SQLite is default; do we ship LMDB option in 1.1 or defer? **Recommend defer to 1.2** unless a pilot customer needs it.
2. **Telemetry opt-out timing.** First-30-days opt-in, then switch — is this consistent with privacy posture? **Founder review with security advisor by Day 14.**
3. **Idempotency key collision window.** Operator deduplicates within a 7-day window. Customers replaying receipts older than 7 days will create duplicates. **Document in production checklist.**
4. **TS SDK supported runtimes.** Node 18+ confirmed; Deno / Bun?  **Defer to 1.2; document Node-only in 1.1.**
